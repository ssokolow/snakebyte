 19      """A queue that maximizes fairness via the following properties: 
 20   
 21       - Users will never be blocked from entering the queue. 
 22       - Users may enqueue as many files as they wish. 
 23       - The number of files one user enqueues will not affect others. 
 24       - Users cannot game the system by adding and removing queue entries. 
 25       - The effect of varying download times on wait times will be minimized. 
 26         **(IN DEVELOPMENT)** 
 27   
 28      It accomplishes this through three techniques: 
 29       - The system maintains one queue per user rather than one global queue 
 30       - Whenever a new file is popped from the queue, it comes from the queue 
 31         of the user who has been waiting the longest in wall-clock time. 
 32       - New users are added to the queue as if they have just received a file, 
 33         which ensures that attempts to game the system will either increase 
 34         their wait time or leave it unaffected rather than decreasing it. 
 35   
 36      At present, it behaves as a simple round-robin queue based on timestamp 
 37      ordering but, because it uses a heap and can take a callback to calculate 
 38      or recalculate a bucket's priority, changes necessary to implement a 
 39      more advanced form of balancing should be minimal with the existing API 
 40      remaining stable. (With the possible exception of adding a new method 
 41      to force a recalculation of a specific bucket's priority once a download's 
 42      total time to completion has become known.) 
 43   
 44      Naturally, it need not only be used for users and lists os files. 
 45   
 46      Its API follows that of the ``dict`` built-in wherever possible, 
 47      though minor variations prevent the methods it implements from being 
 48      drop-in compatible the way ``OrderedDict`` is. 
 49   
 50      The following API incompatibilities with ``dict`` are known: 
 51          - `__len__` returns the total number of items in all buckets, 
 52            not the number of buckets accessed by `__iter__` and `__getitem__`. 
 53          - `pop`'s behaviour is a cross between ``dict.pop``, ``dict.get``, and 
 54            ``dict.popitem``. 
 55   
 56              - ``dict.pop`` may take 1 or 2 arguments, while ``pop`` may take 0 
 57                or 1. 
 58              - ``pop``'s return value is similar to that of ``dict.popitem`` 
 59              - After calling ``pop``, the number of keys in the queue may or may 
 60                not have decreased by 1. With ``dict.pop`` it will always 
 61                decrease by 1. 
 62   
 63      :todo: Decide on an API for getting the number of buckets more efficiently

 65      :todo: Inherit from ``UserDict.DictMixin`` and unit test what it adds. 
 66      :todo: Once I've got the fserve working, come back and add the support for

 68             with those who requested much bigger files.
